git-基本操作（二）

1. 工作区与暂存区

为什么 Git 比其他版本控制系统设计得优秀，因为 Git 跟踪并管理的是修改，而非文件。 git commit 只负责把暂存区的修改提交了。 每次修改，如果不 add 到暂存区，那就不会加入到 commit 中。 主要注意下面三个命令

2. 撤销修改

命令git checkout – readme.txt意思就是，把readme.txt文件在工作区的修改全部撤销，这里有两种情况：

一种是readme.txt自修改后还没有被放到暂存区，现在，撤销修改就回到和版本库一模一样的状态；

一种是readme.txt已经添加到暂存区后，又作了修改，现在，撤销修改就回到添加到暂存区后的状态。

总之，就是让这个文件回到最近一次git commit或git add时的状态。

3. 删除文件

一是确实要从版本库中删除该文件，那就用命令git rm删掉

git checkout其实是用版本库里的版本替换工作区的版本，无论工作区是修改还是删除，都可以“一键还原”。

git-最常用的操作

git版本控制系统，能跟踪文本文件的改动，比如TXT文件，网页，所有的程序代码等等。

1. git init

在本地创建一个空目录，然后用git init把刚刚建的空目录变成Git可以管理的仓库。
.git目录是Git来跟踪管理版本库的.

2. git status

用该命令可以时刻掌握仓库当前的状态

3. git add filename

用该命令把fliename添加到仓库的暂存区。

4. git commit -m “修改了些什么东东”

该命令把暂存区中的文件提交到master

5. git reflog

用该命令查看命令的历史

6. git log

用该命令查看从最近到最远的提交日志

7. git log --pretty=oneline

用该命令查看commit id，在想返回到以前的版本时可以用

8. git log --graph

用该命令查看分支合并图

9. git reset

用该命令回退到之前的版本

10. git diff filename

用该命令查看difference。

11. git diff HEAD -- filename

用该命令查看工作区和版本库里最新版本的区别。

12. git checkout -- filename

用该命令把文件filename在工作区中的修改全部撤销。
（当改乱了工作区某文件中的内容时用上面的命令丢弃工作区中的修改）

13. git reset HEAD filename

当改乱了工作区中的某文件中的内容，且把该文件提交到了暂存区，需要走下面两步去撤销。

14. git rm filename

删除版本库中的文件。

15. git branch dev

创建了一个叫dev的分支

16. git checkout dev

切换到分支dev

17. git checkout -b dev

创建分支dev，并切换到分支dev。

18. git branch

用该命令查看当前所在的分支。

19. git checkout master

切回到master分支

20. git merge dev

把dev分支上的工作成果合并到master分支上

21. git branch -d dev

删除dev分支。

22. git stash

把当前工作现场“储藏”起来，以后可以恢复工作现场，

23. git stash list

查看“储藏”列表

24. git stash pop

恢复工作现场，并把stash内容删除

25. git remote add origin git@git.llsapp.com:scorix/telis-api-test.git

把本地仓库与远程仓库关联起来

26. git push -u origin master

把当前的分支推送到远程库

27. git clone git@git.llsapp.com:scorix/telis-api-test.git

clone代码到本地工作区

28. git pull <远程主机名> <远程分支名>:<本地分支名>

取回远程主机某个分支的更新，再与本地的指定分支合并

29. git rebase

用于把一个分支的修改合并到当前分支

http协议-接口测试基础（二）

1.常用的HTTP请求方式

2. 中介

中介由三种：
代理(Proxy)、
网关(Gateway)
通道(Tunnel)，
代理(Proxy)：一个中间程序，它可以充当一个服务器，也可以充当一个客户机，为其它客户机建立请求。请求是通过可能的翻译在内部或经过传递到其它的 服务器中。一个代理在发送请求信息之前，必须解释并且如果可能重写它。代理经常作为通过防火墙的客户机端的门户，代理还可以作为一个帮助应用来通过协议处 理没有被用户代理完成的请求。
网关(Gateway)：一个作为其它服务器中间媒介的服务器。与代理不同的是，网关接受请求就好象对被请求的资源来说它就是源服务器；发出请求的客户机并没有意识到它在同网关打交道。
网关经常作为通过防火墙的服务器端的门户，网关还可以作为一个协议翻译器以便存取那些存储在非HTTP系统中的资源。
通道(Tunnel)：是作为两个连接中继的中介程序。一旦激活，通道便被认为不属于HTTP通讯，尽管通道可能是被一个HTTP请求初始化的。当被中继 的连接两端关闭时，通道便消失。当一个门户(Portal)必须存在或中介(Intermediary)不能解释中继的通讯时通道被经常使用。
